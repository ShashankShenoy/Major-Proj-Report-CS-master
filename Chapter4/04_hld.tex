\chapter{High Level Design of Maritime Surveillance and Trajectory Prediction}

This chapter details the high-level architecture of the maritime surveillance platform. It discusses key design considerations, the selection of the technology stack, the end-to-end system architecture overview, and the systematic data flow modeling that transforms raw sensor inputs into predictive intelligence.

\section{Design Considerations}

\subsection{General System Considerations}
Designing a surveillance architecture for maritime environments presents unique engineering challenges that are rarely encountered in standard urban traffic monitoring. The sheer scale of the environment means the system must reliably handle vessels of vastly different sizes, ranging from massive cargo ships occupying the entire frame to small fishing skiffs that appear as mere specks on the horizon. Furthermore, the operational environment dictates strict latency constraints; a warning generated several minutes after a vessel deviates from its path is effectively useless. Therefore, ensuring real-time or near-real-time throughput across the entire processing pipeline was the foremost design priority.

Beyond speed, the system was designed with resilience in mind. Port authorities and coastal guards require continuous oversight, meaning the software architecture must gracefully handle interruptions. If an \gls{ais} data stream drops out, a common occurrence due to signal interference or deliberate transponder deactivation, the visual tracking module must be robust enough to maintain target locks autonomously. 

The visual tracking components also mitigate common atmospheric disturbances inherent to maritime regions, such as heavy fog, extreme glare from the water surface, and low-light conditions at dusk. By decoupling the initial bounding-box generation from the long-term track state estimation, the architecture prevents temporary visual degradation from completely collapsing the tracking history. Furthermore, handling high-density maritime traffic, where multiple vessels routinely obscure one another from the camera's perspective, requires an intelligent re-identification subsystem capable of matching visual embeddings even after severe overlapping events.

The proposed maritime surveillance framework demonstrates a hierarchical organisation. Each processing stage performs a dedicated function and passes structured information to subsequent modules. Such a layered design minimizes inter-module dependency and improves maintainability, scalability, and future extensibility of the overall system.

\subsection{Development Methods and Modularity}
To manage the complexity of integrating multiple deep learning models, an iterative development methodology paired with strict modular design principles was adopted. Rather than building a monolithic application where the detection logic is inextricably tangled with the prediction logic, each core function was compartmentalized into distinct micro-services or independent modules.

This modularity serves two critical purposes. First, it allowed for isolated testing during the development phase; the YOLOv8 detection thresholds could be refined without risking breakage in the downstream tracking algorithms. Second, it ensures the system remains future-proof. Because the interfaces between modules are standardized, the underlying models can be swapped out as technology progresses. For example, if a more efficient tracking algorithm surpasses DeepSORT in the future, it can be integrated simply by conforming to the existing bounding-box input and track-ID output formats, leaving the rest of the pipeline untouched.

\section{Architecture Strategies}

\subsection{Technology Stack and Language Selection}
The primary programming language chosen for the development of this architecture was Python. While compiled languages like C++ offer theoretical advantages in raw execution speed, Python is the undisputed industry standard for deep learning research and deployment, largely due to its vast ecosystem of mature, highly optimized libraries. 

Specifically, PyTorch was selected as the foundational machine learning framework. Its dynamic computation graph and native support for CUDA-accelerated GPU processing were indispensable for achieving real-time inference speeds with both the YOLOv8 and \gls{lstm} networks. For handling the intensive task of raw video decoding and image matrix manipulation, OpenCV was utilized extensively. Its underlying C++ implementation ensures that pre-processing tasks, such as resizing frames and applying contrast enhancements, do not become a computational bottleneck. Finally, to render the complex analytical data intuitively for end-users, libraries like Folium and PyQt were integrated, which allow for the creation of interactive, geographically aware dashboards.

The selected technology stack was chosen to balance development flexibility, computational efficiency, and scalability. Python provides rapid prototyping capabilities and extensive support for machine learning applications, while PyTorch enables efficient implementation of deep neural networks. OpenCV facilitates real-time video processing, whereas Folium and PyQt provide intuitive visualization interfaces for maritime situational awareness. Together, these technologies form a cohesive software ecosystem capable of supporting real-time surveillance operations.

\subsection{Pathways for Future Extensibility}
While the current architecture delivers a robust proof-of-concept for vision-based predictive monitoring, the design explicitly accounts for future expansion. Current operations are limited to single-camera, optical video feeds. However, the data ingestion layer was structured to eventually accommodate multimodal sensor fusion. Future iterations are planned to ingest inputs from long-wave infrared (LWIR) cameras for nighttime operations, as well as raw radar returns to provide depth estimation across the Z-axis. Additionally, the tracking logic has been conceptually mapped to support multi-camera spatial hand-offs, which would allow the system to pass a tracked vessel's identity seamlessly from one camera's field of view to the next, enabling continuous monitoring across an entire sprawling port complex.

\section{System Architecture}
The system architecture follows a sequential, feed-forward pipeline. It begins at the data ingestion layer, which continuously pulls frames from the live camera feed and synchronizes them with incoming \gls{ais} telemetry when available. This raw data is then pushed into the inference engine. Here, the YOLOv8 object detector isolates the vessels and generates tight bounding boxes. These localized regions of interest are immediately passed to the DeepSORT tracking module, which uses a pre-trained OSNet re-identification network to extract deep visual features from the vessel's hull, assigning a persistent unique identifier (ID) to each ship.

Once tracked, the system extracts the center coordinates of the bounding boxes frame-by-frame. This sequential coordinate data is transformed mathematically to calculate real-world velocity and heading vectors. This historical movement tensor is then fed into the \gls{bilstm} trajectory forecasting module, which outputs a predicted pathway for the next several minutes. The final layer of the architecture is the presentation module, which overlays the current bounding boxes, IDs, and predicted trajectory arcs directly onto the live video feed, while simultaneously plotting the data onto a live geospatial map.

To ensure the seamless operation of this pipeline, several architectural mechanisms operate in the background. First, the data ingestion layer must handle the asynchronous nature of the inputs. While the camera feed provides high-frequency visual data (typically 30 frames per second), \gls{ais} telemetry is transmitted at much lower, variable intervals depending on a vessel's speed and class. The system employs a spatio-temporal alignment buffer that interpolates missing \gls{ais} data points to match the video frame rate, ensuring that the downstream fusion engine receives synchronized multi-modal tensors.

Furthermore, the inference engine is designed to maintain target locks even under severe maritime conditions. When vessels pass behind obstacles or occlude one another, the DeepSORT module relies heavily on the OSNet feature embeddings. Rather than depending solely on spatial overlap metrics, the system compares the visual fingerprint of a vessel before and after occlusion, allowing it to recover the correct identifier and seamlessly continue the track. This prevents track fragmentation, which would otherwise corrupt the historical movement tensor fed into the prediction module.

The \gls{bilstm} trajectory forecasting module itself operates on a sliding window protocol. It continuously ingests the most recent coordinate vectors while discarding older data beyond a predefined temporal threshold. By analyzing the sequence bidirectionally, the network captures both immediate past momentum and broader historical trends, allowing it to generate highly accurate, dynamic predictions. The architecture isolates this heavy computation from the main rendering thread to prevent visual stuttering on the operator's end.

Finally, the presentation module is optimized for low-latency delivery. It utilizes hardware-accelerated rendering to overlay bounding boxes, alphanumeric IDs, and the curved, multi-minute predicted trajectory arcs.

\section{Summary}
This chapter established the foundational High Level Design and architectural framework of the proposed maritime surveillance platform. By prioritizing strict modularity and utilizing an asynchronous, feed-forward pipeline, the architecture guarantees that computationally intensive deep learning models, such as YOLOv8, DeepSORT, and the \gls{bilstm} forecasting networks, can operate synchronously without introducing critical bottlenecks. The selection of a robust Python-based technology stack, paired with PyTorch and OpenCV, ensures both real-time performance and long-term scalability. Furthermore, the detailed Data Flow Diagrams (Levels 0 through 2) provided a clear, logical roadmap demonstrating exactly how raw, uncoordinated sensor inputs (video frames and \gls{ais} telemetry) are systematically fused, transformed into historical movement tensors, and ultimately refined into actionable collision-avoidance intelligence for end-users. This structured approach ensures the system remains reliable under challenging maritime conditions while leaving explicit pathways open for future multimodal sensor integration.
